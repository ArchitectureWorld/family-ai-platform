# Family AI Platform：DIY 硬件终端开发方案

## 1. 终端定位

DIY 硬件不是 Web 或移动端的缩小版，而是面向具体家庭场景的轻量入口。

核心定位：

```text
低门槛
低操作成本
语音优先
场景化
能力受限
```

第一版重点是让现有 ESP32 类硬件能够作为个人助理或家庭公共入口接入平台。

---

## 2. 终端分类

DIY 终端至少分为两类。

### 2.1 个人型终端

示例：

- 个人桌面终端；
- 儿童终端；
- 老人终端；
- 个人佩戴设备。

绑定关系：

```text
device_id
→ person_id
→ HomeChatStream
→ Personal Assistant
```

### 2.2 家庭共享终端

示例：

- 客厅音箱；
- 公共智能屏；
- 家庭控制终端。

绑定关系：

```text
device_id
→ family_id
```

使用时再识别或选择 Person。

---

## 3. 终端能力模型

DIY 设备必须注册能力。

```json
{
  "terminal_type": "diy_voice_screen",
  "capabilities": {
    "screen": "small",
    "touch": true,
    "voice_input": true,
    "voice_output": true,
    "text_input": false,
    "camera": false,
    "file_upload": false,
    "local_wake_word": true
  }
}
```

不能假设所有 DIY 设备都有：

- 屏幕；
- 触摸；
- 扬声器；
- 麦克风；
- 摄像头；
- 常在线网络。

---

## 4. Chat 体验

个人型 DIY 终端默认映射到 Chat。

典型流程：

```text
唤醒
→ 录音
→ 语音识别
→ 发送到 HomeChatStream
→ Agent 回复
→ 语音播报 / 小屏显示
```

输出必须适配：

- 简短；
- 口语化；
- 一次只表达一个重点；
- 复杂内容提示用户到 Web 或移动端查看；
- 涉及执行时进行明确确认。

---

## 5. Work 能力

第一版 DIY 不需要完整 Work 页面。

只支持：

- 查询某个 Work 状态；
- 将想法记录到某个 Conversation；
- 新建轻量 Conversation；
- 接收待确认事项；
- 执行确认 / 拒绝；
- 提醒用户转到移动端或 Web 继续。

示例：

```text
“把这个想法放到 family-ai-platform 那个工作里。”
```

系统需要将语音解析并关联到指定 Conversation。

---

## 6. 身份策略

### 个人型设备

第一版推荐管理员预绑定，不要求设备上输入手机号。

```text
管理员后台生成绑定码
→ 设备输入或扫码
→ 绑定 person_id
```

### 家庭共享设备

身份策略按阶段推进。

第一版：

```text
屏幕选择成员
或
口头说出成员名并确认
```

后续再考虑：

- 声纹；
- 手机邻近；
- 可穿戴设备辅助；
- 人脸；
- 多因素识别。

身份不明确时，不允许访问私人 Work。

---

## 7. 公共与私人边界

家庭共享终端默认可处理：

- 家庭公共提醒；
- 晚餐；
- 家庭日程；
- 公共设备控制；
- 非敏感问答。

默认不可直接播报：

- 私人 Work；
- 私人消息；
- 私人记忆；
- 敏感日程；
- 账号安全信息。

需要私人内容时，要求二次确认或转到个人终端。

---

## 8. 唤醒与对话状态

第一版建议使用：

```text
唤醒词
→ 单轮指令
→ 必要时短时连续对话
→ 自动结束
```

不能将设备麦克风长期上传音频。

对话状态需要明确指示：

- 正在监听；
- 正在识别；
- 正在处理；
- 正在播报；
- 已结束；
- 出错。

可通过 LED、屏幕图标和声音提示实现。

---

## 9. 设备注册与管理

Web 管理端需要支持：

- 创建设备；
- 生成绑定码；
- 查看在线状态；
- 查看固件版本；
- 设置归属；
- 设置位置；
- 配置能力；
- 冻结；
- 解绑；
- 重置凭证。

设备必须使用独立凭证，不能硬编码家庭统一密钥。

---

## 10. 连接协议要求

第一版设备接入协议至少支持：

```text
设备认证
心跳
能力上报
音频上传
文本消息
事件消息
状态下发
音频播放
OTA 状态
```

推荐设备与平台之间通过统一网关，而不是直接连接 Hermes。

```text
DIY Device
→ Device Gateway
→ Identity / Device Resolve
→ Chat or Work
→ Personal Assistant
```

---

## 11. 本地与云端职责

### 设备本地

适合：

- 唤醒词；
- 录音；
- LED / 屏幕；
- 按键；
- 网络管理；
- 简单缓存；
- 播放。

### 平台或局域网服务

适合：

- 语音识别；
- Agent；
- 语音合成；
- 身份解析；
- Chat / Work；
- 记忆；
- 设备管理。

第一版不要求在 ESP32 上运行复杂 Agent。

---

## 12. 网络异常

设备需支持：

- 无网提示；
- 自动重连；
- 录音发送失败提示；
- 不重复执行指令；
- 命令唯一 ID；
- 离线时只提供有限本地功能；
- 恢复后不上报已经过时的危险指令。

---

## 13. 安全

必须包括：

- 设备独立凭证；
- 传输加密；
- 远程吊销；
- 恢复出厂；
- 敏感操作确认；
- 设备来源记录；
- 公共终端隐私保护。

家庭控制类动作需要根据风险分级。

---

## 14. 第一版设备建议形态

结合现有硬件，第一版可以采用：

```text
ESP32
+ 麦克风
+ 扬声器
+ 小屏
+ LED
+ 按键
```

第一版目标不是完整智能音箱，而是验证：

- 设备注册；
- 个人绑定；
- 语音问答；
- Chat 连续性；
- 简单确认；
- 跨端查看。

---

## 15. 第一版范围

### 必须

- 设备注册；
- 管理员绑定；
- 个人型终端；
- 语音输入；
- 语音输出；
- Chat 接入；
- LED / 小屏状态；
- 设备在线状态；
- 基础安全；
- 跨端消息可见。

### 暂缓

- 自动声纹；
- 多人自然切换；
- 连续全双工语音；
- 复杂 Work 浏览；
- 本地大模型；
- 家庭设备自动化控制；
- 多房间音频网络。

---

## 16. 体验验收

- 设备可以安全注册和解绑；
- 个人型设备能进入正确 Person 的 Chat；
- 设备发出的消息在 Web / 手机可见；
- Agent 回复能以适合硬件的形式输出；
- 网络中断时用户得到明确反馈；
- 共享设备不会泄露私人 Work；
- 重复重试不会重复执行；
- 复杂内容能自然引导用户到完整终端继续。
